## Version Contract

**Platform contract:** [Semantic Versioning 2.0.0](https://semver.org/) applies to a declared public API. Define that API before assigning meaning to a bump.

SemVer examples from `1.0.0`, evaluated against that declared public API:

- `1.0.0 -> 1.0.1`: backward-compatible defect/security fix.
- `1.0.0 -> 1.1.0`: backward-compatible functionality or deprecation.
- `1.0.0 -> 2.0.0`: incompatible public API change.
- `2.0.0-rc.1 -> 2.0.0`: promote a tested release candidate; prerelease identifiers have lower precedence than the normal release.
- `1.0.0+build.7`: build metadata identifies provenance but does not change SemVer precedence.

**Kit/project policy:** explicitly decide whether operator actions, configuration, persisted data, required dependencies, supported runtimes and observable behavior are part of the plugin's declared public product contract. If they are, keep old consumers/config/data valid for PATCH/MINOR releases and use MAJOR for an incompatible contract change. A dependency update is not automatically breaking; judge the public contract it changes.

For `0.y.z`, SemVer treats the public API as unstable; still publish and document a project-specific compatibility policy. Minecraft/Paper support changes do not mechanically dictate a bump: decide whether dropping a runtime breaks the plugin's declared public product contract.

Released version contents are immutable. If `1.0.0` bytes or metadata are wrong, issue `1.0.1` or a documented replacement/yank; never overwrite `1.0.0` with different bytes.

Keep one version source and verify it matches:

- Git tag/release title;
- Gradle project/publication version;
- `plugin.yml` and/or `paper-plugin.yml`;
- JAR filename/manifest where used;
- SBOM, checksums, changelog and marketplace version.

## Release Evidence Bundle

Produce these side by side:

```text
ExamplePlugin-<version>.jar       exact tested artifact
ExamplePlugin-<version>.jar.sha256
ExamplePlugin-<version>.spdx.json or .cdx.json
CHANGELOG excerpt/release notes
compatibility-matrix.md or .json
LICENSE + required third-party notices
build provenance/attestation when available
```

The SHA-256 file should identify the artifact unambiguously. Examples:

```text
Get-FileHash .\build\libs\ExamplePlugin-<version>.jar -Algorithm SHA256
sha256sum build/libs/ExamplePlugin-<version>.jar
```

**Kit recommendation:** an SBOM includes bundled/shaded components and runtime-provisioned libraries, with relationship/scope, resolved version, license and hash when available. Use a standard such as [SPDX](https://spdx.dev/) or [CycloneDX](https://cyclonedx.org/). Scan the final JAR too; a dependency graph alone can miss relocated or generated bytes.

Release notes should state:

- user-visible changes, fixes, deprecations and removals;
- required Minecraft/Java/platform changes;
- configuration/schema migrations and backup steps;
- public API and integration changes;
- new/removed runtime libraries or permissions;
- security fixes without exploit instructions where disclosure is coordinated;
- known issues, fallback and rollback path.

## Compatibility Matrix

Do not use “works on latest.” Attach the matrix to the artifact checksum:

| Dimension | Declared value | Evidence/status |
|---|---|---|
| Artifact | filename + SHA-256 | two clean builds / final inspection |
| Minecraft | exact IDs or tested range | exact server integration runs |
| Java | minimum/maximum claimed runtime + emitted bytecode | toolchain and launch runs |
| Paper | exact tested builds | pass/fail + test run link |
| Purpur | exact tested builds and capability fallback | pass/fail + test run link |
| Folia | exact tested builds; descriptor flag | global/region/entity/async ownership tests |
| Descriptor | `plugin.yml`, `paper-plugin.yml`, or both | parsed resource + startup path |
| NMS | none or exact adapter/build | compile/linkage/runtime probe |
| Dependencies | required/optional versions and provider | absence/mismatch tests |
| Storage | engine/driver/pool/schema versions | migration/outage/recovery tests |
| Clients/packs | exact variants if relevant | clean-client load/degradation |
| Reload | unsupported restart-only or exact tested scope | lifecycle test |

Keep `available`, `tested`, `supported`, and `compatible` separate. A marketplace version selector is metadata, not runtime evidence.

## Build-Once Pipeline

Recommended release stages:

1. **Prepare:** select SemVer, target matrix, source commit/tag, changelog and migration/backup plan. Confirm author/nametag `SalyVn / Salyyy`.
2. **Resolve:** use the committed Gradle Wrapper/toolchain; enforce dependency locks and verification; reject unreviewed dynamic/changing inputs.
3. **Check:** compile, unit/static tests, descriptor/config/JSON validation, license/security checks and public API compatibility checks.
4. **Integrate:** start exact Paper/Purpur/Folia targets; exercise lifecycle, ownership, database, missing dependency, NMS/pack and load tests.
5. **Build once:** create one release JAR in a clean CI job. Inspect contents and compute SHA-256.
6. **Prove:** rebuild in an isolated clean directory, compare hashes, generate SBOM/notices/provenance, and bind all evidence to the artifact hash.
7. **Promote:** pass the stored artifact to a secret-scoped publishing job. Do not run `build` again in channel-specific jobs.
8. **Publish:** upload identical bytes and matching metadata to approved channels after manual/environment approval when appropriate.
9. **Verify:** download each public asset, recompute SHA-256, install on a clean declared target and verify the listing/download path.
10. **Observe:** monitor startup errors, migration failures and support channels; retain the artifact/evidence for rollback and audit.

When CI cannot preserve an artifact between jobs with integrity, publish from one gated job after all tests and record the hash before each upload. Never let a marketplace plugin silently transform or rebuild the JAR.

## CI Secrets And Supply Chain

Split CI trust zones:

```text
untrusted pull request -> build/test only, no publishing secrets
trusted tag/release -> reproduce or promote tested artifact
protected environment -> checksum gate + scoped marketplace upload
```

- Store Hangar/Modrinth/Spigot/GitHub credentials in CI secret/environment storage, not Gradle files, source, caches, artifacts or logs.
- Use one token per service/repository with the smallest publication scope; rotate and revoke independently. Prefer short-lived/OIDC credentials when a platform supports them.
- Give the build job no marketplace secrets. Release them only to a protected publishing job after tag, branch, approval and checksum conditions pass.
- Never execute or check out untrusted fork code in a privileged `pull_request_target`-style context. Fork pull requests must not receive secrets.
- Pin third-party CI actions/plugins to reviewed immutable commit SHAs where supported. Validate the Gradle Wrapper and dependency verification metadata.
- Avoid shell tracing and printing environment/config. Mask values, sanitize exception output and do not upload debug archives containing environment state.
- Set explicit workflow permissions; GitHub's `GITHUB_TOKEN` should have only the permissions used by that job.
- Keep a dry-run/staging channel and test token expiry, permission denial, duplicate version, network timeout and partial multi-channel publication.

See GitHub's [secrets guidance](https://docs.github.com/en/actions/concepts/security/secrets) and [artifact attestations](https://docs.github.com/en/actions/how-tos/secure-your-work/use-artifact-attestations/use-artifact-attestations).

## Publishing Channels

### GitHub Releases

- Create the release from the reviewed source tag/commit; do not move a released tag.
- Attach the exact JAR, SHA-256, SBOM, notices, compatibility matrix and release notes.
- Use release provenance/attestations when appropriate and verify them with the documented client.
- Make the primary download obvious; mark prerelease/draft status accurately.
- After upload, download and hash the asset. Link issues/security policy and support scope.

Use GitHub's [release management](https://docs.github.com/en/repositories/releasing-projects-on-github/managing-releases-in-a-repository) and [artifact attestation](https://docs.github.com/en/actions/how-tos/secure-your-work/use-artifact-attestations/use-artifact-attestations) documentation.

### Hangar

- Create the project/channel and token with only the required version-publication permission.
- Keep `HANGAR_API_TOKEN` in the protected CI environment described by Hangar.
- Set exact Paper/Velocity/Waterfall platform versions and plugin dependencies; do not copy a server range from another marketplace without evidence.
- Point publication at the stored release JAR and matching changelog/channel; do not rebuild in `publishPluginPublicationToHangar` without verifying the produced input hash.
- Verify the public download hash, listing dependencies, channel and compatibility metadata.

Follow the current [Hangar auto-publishing guide](https://docs.papermc.io/misc/hangar-publishing/) for the pinned publishing plugin; its examples are not permanent version recommendations.

### Modrinth

- Create a version with correct project, version number/name, release type, loaders, game versions, dependencies and changelog.
- A non-draft version requires an accepted uploaded file; upload the already-tested JAR as the primary file.
- Keep the token scoped and secret, detect duplicate versions, and verify the returned file hash/download.
- Do not infer Paper/Purpur/Folia support merely from a loader label; keep the detailed matrix in release notes/docs.

Use Modrinth's current [create-version API](https://docs.modrinth.com/api/operations/createversion/) or an audited publisher that preserves the exact artifact.

### SpigotMC

- Review the current [resource guidelines](https://www.spigotmc.org/wiki/resources/) and account/resource rules before each release.
- Upload the same JAR manually unless an explicitly supported publishing mechanism is available and authorized.
- State tested Minecraft/Java/server range, dependencies, commands/permissions, install/upgrade/migration steps and support link accurately.
- Do not advertise Paper/Purpur/Folia/NMS compatibility that the matrix did not test. Verify the downloaded file hash after approval/publication.
- Never automate through scraped sessions, browser cookies or stored personal credentials.

## Post-Release And Rollback

- Record listing URLs, version IDs, publication timestamps and observed SHA-256 for every channel.
- Smoke-test a clean install and an upgrade from the oldest supported data/config schema.
- If distribution metadata is wrong but bytes are correct, amend metadata only where the platform preserves artifact identity and audit history.
- If bytes, dependencies or migrations are wrong, stop/mark the release unsupported where possible and publish a new SemVer version. Do not replace the old bytes.
- For database incidents, stop writes, preserve evidence, restore through the tested engine-aware backup plan, and run a forward repair migration. A JAR rollback alone may not reverse schema/data changes.
- Revoke tokens immediately after suspected exposure and audit channel/version history.

## Gate Checklist

- [ ] Public compatibility contract and SemVer bump approved.
- [ ] Exact Minecraft, Java, Paper, Purpur, Folia, dependency and storage matrix recorded.
- [ ] Wrapper, plugins, dependencies, locks and verification metadata reviewed.
- [ ] Descriptor version/authors (`SalyVn / Salyyy`) and YAML/JSON resources validated.
- [ ] Unit, integration, ownership, database, lifecycle, performance and failure tests pass.
- [ ] Final JAR inspected; no secrets, mutable production data, server APIs or Minecraft binaries bundled.
- [ ] Licenses/notices, vulnerability review, SBOM and compatibility matrix complete.
- [ ] Two clean builds match; release JAR SHA-256 recorded.
- [ ] Changelog, migrations, backups, upgrade and rollback instructions complete.
- [ ] Publishing jobs use protected scoped secrets and the stored artifact, without rebuilding.
- [ ] GitHub, Hangar, Modrinth and SpigotMC metadata match the artifact and evidence.
- [ ] Public downloads re-hashed and clean-install/upgrade smoke tests pass.

Unmet gates remain `UNVERIFIED`; do not convert them into release claims.
